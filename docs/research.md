# Sui 链上异常监控研究笔记

## 1. 产品目标
面向 Sui 项目方做“黑客事件前置预警”，核心不是事后复盘，而是尽可能在资金真正大规模流出前识别异常信号：
- 管理权限异常使用
- 包升级异常
- 资金金库/热钱包异常流出
- 某个包/对象在极短时间内被异常高频访问
- 失败交易、探测交易、权限绕过尝试骤增

## 2. 从官方资料确认到的关键事实
1. Sui 官方文档已明确说明：JSON-RPC 将在 2026 年 7 月前迁移到 gRPC 或 GraphQL RPC。
2. 官方推荐生产环境不要直接依赖公共 RPC，因为公用端点限流严格；商业化产品应使用自建或商业节点服务。
3. Sui 官方提供两条适合监控系统的数据接入路线：
   - GraphQL RPC：适合快速落地查询、分页、对象/交易/事件聚合查询。
   - gRPC / 自定义 Indexer：适合高吞吐、低延迟、连续流式消费。
4. 官方自定义索引框架 `sui-indexer-alt-framework` 采用 checkpoint 驱动，说明“按 checkpoint 做增量处理”是官方认可的工程范式。

## 3. 适合黑客预警的检测面
### A. 权限/升级面
- 监控关键 package version 是否变化
- 监控升级交易最近 sender 是否在 allowlist 中
- 监控高危管理函数（如 emergency_withdraw / set_admin / mint / pause / upgrade）是否由非授权地址调用

### B. 资金面
- 监控 treasury / vault / admin wallet 的负向 balance change
- 对不同 coin type 设置不同阈值
- 同时记录触发交易调用了哪些 package/module/function，帮助快速判断是否为正常业务流出

### C. 交互面
- 某个关键 package 在短时间内交易数暴涨
- 关键 package 的 unique sender 在短时间内暴涨
- 关键 package 失败交易短时激增，通常意味着攻击者在试错、探测或撞参数

### D. 后续应补强的高级能力
- 对 shared object 的版本变化做精细画像
- 对价格敏感协议增加预言机/池子储备异常监控
- 引入历史基线（小时、日、周）做动态阈值而不是只用静态阈值
- 结合链外情报（审计公告、漏洞披露、社区告警地址）做综合评分

## 4. 为什么当前版本先做“规则引擎 + checkpoint 增量扫描”
因为这是最稳的 MVP 路线：
- 上手快，可立刻开始接入项目方真实地址/包
- 可解释性强，适合安全团队接警
- 后续可以平滑替换数据源：
  - 当前：GraphQL 轮询 checkpoint
  - 下一步：gRPC 流式消费
  - 再下一步：自建 indexer + 统计/ML 异常评分

## 5. 商业化建议
- 不要把产品定义成“黑客一定能提前发现”，而要定义成“协议安全异常早期信号平台”
- 商业版至少要有：
  1. 多项目租户隔离
  2. 规则模板市场
  3. Webhook / Slack / Telegram / Email 告警
  4. 事件归并与去重
  5. 历史回放与误报标注
  6. 审计/值班团队协同视图

## 6. 当前仓库实现范围
当前仓库已先实现：
- GraphQL checkpoint 轮询
- package 升级检测
- 资金异常流出检测
- 高危函数越权调用检测
- 交易热度突增检测
- 失败交易突增检测
- HTTP 查询接口
- Webhook/控制台告警

后续建议优先做：
1. Postgres 持久化
2. 告警去重与升级策略
3. gRPC 数据源适配器
4. 前端控制台
5. 多租户与权限系统
