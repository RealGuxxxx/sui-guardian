# 生产数据清理与真实数据模式切换设计

## 目标

将 `sui-guardian` 从“可展示实验/演练数据的开发态”切换为“默认仅展示真实业务数据的生产态”。

本次改造必须同时满足以下要求：

- 彻底移除界面、状态文件、默认配置、接口链路中的 demo / generated / sample 展示数据
- 保留演练验证功能模块及其相关代码与脚本
- 默认运行时仅连接真实数据源，并展示真实项目数据
- 当未配置真实项目时，系统返回空结果或真实空态，不生成任何示例数据
- 输出数据清理报告与配置变更记录

## 非目标

- 不删除 `DeFiHackLabs-main` 研究资料
- 不删除 `scripts/run_vuln_defi_lab.py`、`scripts/run_defi_range_lab.py` 等演练脚本
- 不移除测试代码中的测试样本；测试样本仅限测试进程内使用，不得进入运行时展示链路
- 不新增新的业务规则或新的检测器能力

## 当前问题

当前仓库同时承载三类内容：

1. 真实监控平台代码
2. 演练验证配置与脚本
3. 已生成的实验状态文件与默认演示运行方式

主要风险点如下：

- `config/generated-defi-range.yml`、`config/generated-vuln-defi-lab.yml` 可被直接作为启动配置，容易让运行界面展示实验项目
- `.data/generated-*.json`、`.data/generated-*.log` 会保留实验运行结果，容易被误认为真实业务数据
- 前端文案仍使用“实验室 / 靶场”等名称时，会让生产模式边界不清晰
- 如果继续用实验配置启动，即使数据来自真实链上，也不属于真实业务项目数据

## 设计原则

- 默认生产优先：默认启动、默认配置、默认页面都只服务真实项目
- 演练能力隔离：演练配置和脚本保留，但不参与默认运行
- 空态优先于假态：没有真实数据时显示空态，不生成占位样本
- 单一事实来源：前端仅消费后端真实接口，后端仅读取真实状态文件和链上扫描结果
- 可审计：清理内容、保留内容、配置变更必须可记录

## 方案

### 方案 A：彻底删除所有实验配置和状态

优点：

- 最彻底
- 误用概率最低

缺点：

- 破坏演练验证模块的直接可用性
- 后续回归验证需要重新搭建演练环境

### 方案 B：保留演练模块，清空默认展示链路

优点：

- 满足生产态纯净要求
- 保留演练验证能力
- 改动边界清晰

缺点：

- 仍需维护“生产配置”和“演练配置”的边界

### 方案 C：仅隐藏前端演示内容

优点：

- 改动最小

缺点：

- 后端和状态层仍可能污染真实展示
- 不满足“彻底移除 demo 数据”的要求

## 结论

采用方案 B。

即：

- 保留演练验证模块代码与脚本
- 移除默认展示链路中的实验配置、实验状态、实验文案
- 将默认运行模式切回真实项目配置

## 具体改造

### 1. 配置层

- 保留 `config/default.yml` 作为默认启动配置
- 保持 `default.yml` 中 `projects: []`，不内置任何示例项目
- 将 `config/generated-defi-range.yml`、`config/generated-vuln-defi-lab.yml` 视为演练专用配置，不再作为默认启动演示路径
- 新增或更新真实项目配置说明，指导用户使用 `projects.example.yml` 填写真实地址

### 2. 状态与数据文件层

- 删除 `.data/generated-defi-range-state.json`
- 删除 `.data/generated-vuln-defi-lab-state.json`
- 删除 `.data/generated-defi-range-monitor.log`
- 删除 `.data/generated-vuln-defi-lab-monitor.log`
- 保留真实运行使用的 `.data/state.json`
- 保留其他真实业务状态文件；若文件内容来源于实验配置，则一并清理

### 3. 运行入口层

- 默认启动命令继续使用 `config/default.yml`
- 不再在交付说明、界面验证流程、默认运行演示中使用 generated 配置
- 演练脚本只能通过显式命令触发，不参与默认启动链路

### 4. API 层

- 检查 `MonitorService`、`server.ts` 是否存在 mock/sample/generated fallback
- 所有接口仅允许返回：
  - 当前真实配置对应的扫描结果
  - 当前真实状态文件中的真实记录
  - 空数组 / 空态对象
- 不允许在无数据时构造示例 metrics、incident、asset 或 alert

### 5. 前端层

- 删除或替换所有会让用户误解为 demo 的文案
- 空态统一改为“未配置真实项目”或“暂无真实数据”
- 不在页面中出现“靶场”“实验室”“demo”“generated”等来源于演示环境的项目名，除非用户手动以演练配置启动

### 6. 演练验证模块

- 保留实验配置文件与脚本
- 保留相关代码路径与执行能力
- 明确其用途为：
  - 数据完整性验证
  - 监控规则回归验证
  - 生产链路演练
- 但默认运行、默认展示、默认状态文件不得引用这些演练配置

## 数据流要求

生产模式下的数据流必须满足：

1. 前端调用 `/api/*`
2. API 从 `MonitorService` 读取运行时状态
3. `MonitorService` 只读取真实配置与真实状态文件
4. 真实状态由链上扫描结果写入
5. 若无真实项目配置，则返回空结果

任何一步都不得注入演示数据、样本记录或 mock 返回值。

## 验证方案

### 运行时验证

- 使用默认配置启动项目
- 确认页面不出现实验项目名
- 确认接口返回空态或真实项目数据
- 确认 `.data` 中不再读取 generated 状态文件

### 配置验证

- 检查默认启动路径是否仍指向 `config/default.yml`
- 检查文档与命令示例是否不再默认引导到 generated 配置

### 数据链路验证

- 验证 `/api/config`
- 验证 `/api/metrics`
- 验证 `/api/incidents`
- 验证 `/api/assets`
- 验证这些接口在无真实项目配置时均返回真实空态

## 交付物

- 代码改动
- 数据清理报告
- 配置变更记录

## 风险与缓解

- 风险：误删演练验证所需脚本
  - 缓解：只清理运行时展示数据，不删除脚本与配置模板

- 风险：清理状态文件后页面为空，被误认为异常
  - 缓解：提供明确空态提示，说明尚未配置真实项目

- 风险：仍有文档或命令默认指向 generated 配置
  - 缓解：同步更新 README、运行说明和相关文档
